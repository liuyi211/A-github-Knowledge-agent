## 1. 可行性评估

### 1.1 高可行性（直接实现）

| 功能 | 理由 |
|------|------|
| DM 过滤 | `event_handler.py` 已有 `chat_id` 字段，只需判断 `chat_type` |
| /omka bind | `command_router.py` 已有命令框架，新增 handler 即可 |
| /omka run | 同上，需增加管理员权限检查 |
| conversation_messages 表 | 复用现有 SQLModel 模式，与 `FeishuEventLog` 类似 |
| agent_runs 表 | 同上 |
| feishu_direct_conversations 表 | 同上 |
| Agent 配置 UI | 复用 SettingsPage 现有 Field/Section 组件模式 |

### 1.2 中等可行性（需要设计）

| 功能 | 挑战 | 解决方案 |
|------|------|----------|
| SimpleKnowledgeAgent | 需要 LLM 多轮对话能力 | 扩展现有 `LLMClient`，增加 conversation 支持 |
| ContextBuilder | 需要从 DB 检索相关内容 | 使用 SQLModel 查询 + 关键词匹配（非向量） |
| 推送到绑定单聊 | 需要查询 `feishu_direct_conversations` | 修改 `daily_job.py` 推送逻辑 |

### 1.3 需要注意的风险

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| LLM 调用成本 | Agent 每次对话都调用 LLM | 限制上下文长度，设置超时 |
| 对话历史存储 | `conversation_messages` 无限增长 | 设置 TTL，定期清理 |
| 事件去重内存 | 当前使用内存 set，重启丢失 | 可接受（飞书有重试机制） |
| 群聊消息过滤 | 需要正确判断 `chat_type` | 飞书事件中 `message.chat_type` 字段 |

**关键设计**:

1. **base.py** - 定义 Agent 接口
```python
class AgentContext(BaseModel):
    user_message: str
    conversation_id: str
    user_external_id: str
    recent_messages: list[dict] = []
    digest_items: list[dict] = []
    knowledge_items: list[dict] = []
    candidate_items: list[dict] = []
    user_profile: dict = {}

class AgentResponse(BaseModel):
    answer: str
    used_context: list[dict] = []
    suggested_actions: list[str] = []

class BaseAgent(ABC):
    @abstractmethod
    async def answer(self, context: AgentContext) -> AgentResponse:
        raise NotImplementedError
```

2. **context_builder.py** - 从 DB 检索上下文
- 查询 `CandidateItem`（pending，按 score 排序）
- 查询 `KnowledgeItem`（按 created_at 排序）
- 读取最新 Digest 文件
- 查询 `ConversationMessage`（最近 N 条）
- 使用关键词匹配（非向量检索）

3. **simple_knowledge_agent.py** - 核心 Agent
- 继承 `BaseAgent`
- 使用扩展的 `LLMClient`（增加 conversation 支持）
- 拼接系统 Prompt + 用户 Prompt + 上下文
- 返回 `AgentResponse`

4. **prompts.py** - Prompt 模板
- 系统 Prompt：定义 OMKA 角色和规则
- 用户 Prompt 模板：包含用户问题 + 上下文

## 4. 验收标准（简化版）

| # | 标准 | 验证方式 |

| 原始计划 | 实际调整 | 原因 |

## 6. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| LLM 调用超时 | 中 | Agent 回复失败 | 设置 60s 超时，返回兜底消息 |
| 飞书事件重复 | 低 | 重复回复 | 已有 event_id 去重机制 |
| 对话历史过大 | 中 | DB 存储增长 | 设置 TTL，定期清理 |
| 群聊误触发 | 低 | 资源浪费 | 严格判断 chat_type |
| Agent 回答质量差 | 中 | 用户体验差 | 限制上下文，明确 Prompt 规则 |
