# OMKA MVP 可行性分析报告

> 基于 `omka_MVP_plan.md` 的理解与分析
> 日期：2026-04-27

---

## 一、项目理解

OMKA（Oh My Knowledge Assistant）是一个面向个人用户的**智能知识助手**。MVP 阶段的核心目标是：

1. **自动采集**：每天定时从 GitHub 抓取用户关注仓库和关键词搜索的新仓库
2. **统一处理**：将抓取结果转换为标准化数据结构
3. **智能筛选**：通过去重、打分、排序筛选出高价值内容
4. **个性推荐**：基于用户兴趣画像生成推荐理由
5. **知识沉淀**：生成每日 Markdown 简报，支持用户收藏入库

**核心闭环**：
```
GitHub 信息源 → 自动抓取 → 统一结构化 → 去重候选池 → 个性化排序 → 摘要推荐 → 每日简报 → 收藏入库
```

## 二、技术方案可行性分析

### 2.1 技术栈评估

| 模块 | 推荐技术 | 可行性 | 说明 |
|---|---|---|---|
| 后端框架 | FastAPI | ✅ 高 | Python 生态成熟，轻量高效，适合 MVP |
| 定时任务 | APScheduler | ✅ 高 | Python 标准定时方案，支持 Cron 表达式 |
| HTTP 客户端 | httpx | ✅ 高 | 支持异步、HTTP/2，性能优秀 |
| 数据结构 | Pydantic | ✅ 高 | 类型安全、序列化方便、与 FastAPI 天然集成 |
| ORM | SQLModel / SQLAlchemy | ✅ 高 | 成熟稳定，文档丰富 |
| 数据库 | SQLite | ✅ 高 | 零配置、单文件、MVP 阶段完全够用 |
| GitHub API | GitHub REST API | ✅ 高 | 官方 API，文档完善，配合 Token 额度充足 |
| LLM 摘要 | OpenAI/Qwen/本地模型 | ⚠️ 中 | 需要 API Key，有成本；本地模型需 GPU |
| 输出格式 | Markdown | ✅ 高 | 简单直观，无需额外依赖 |

**结论**：技术栈选型成熟可靠，所有组件都有稳定的社区支持，技术风险较低。

### 2.2 GitHub API 可行性

MVP 需要调用 3 类 GitHub API：

1. **GET /repos/{owner}/{repo}** — 获取仓库基础信息
2. **GET /repos/{owner}/{repo}/releases?per_page=1** — 获取最新 Release
3. **GET /search/repositories** — 关键词搜索新仓库

**API 限制分析**：
- GitHub REST API 对认证用户有 **5,000 requests/hour** 的额度
- MVP 每天抓取：假设 10 个仓库 + 5 个关键词 = 约 15-20 次请求/天
- 额度余量非常充足，即使后续扩展到 100 个仓库也足够

**风险点**：
- Search API 有单独的频率限制（10 requests/minute）
- 需要合理限速，避免触发 403/429

**缓解方案**：
- 实现指数退避重试
- 全局并发控制在 2-5
- 保存 ETag/Last-Modified 支持条件请求

## 三、架构设计评估

### 3.1 分层架构合理性

计划中的 6 层架构清晰且合理：

```
用户配置层 → 调度层 → 信息源接入层 → 规范化层 → 候选池层 → 排序层 → 摘要层 → 输出层
```

**优点**：
- 职责分离清晰，每层只关注一件事
- 便于后续扩展（如增加 RSS Connector 只需实现统一接口）
- 数据流单向，易于调试和排错

**建议**：
- Pipeline 层之间使用明确的数据模型（RawItem → NormalizedItem → CandidateItem）
- 每层增加异常处理，避免单点失败影响全局

### 3.2 数据模型设计

**5 层数据模型**：
- SourceConfig：用户配置
- RawItem：原始抓取数据（保留完整 JSON）
- GitHubRepoData / GitHubReleaseData：GitHub 事实模型
- NormalizedItem：统一知识条目（核心）
- CandidateItem：候选推荐条目

**评估**：
- 分层合理，便于追溯和排错
- NormalizedItem 作为统一接口，扩展性好
- 建议增加数据库索引（如 url、content_hash）提升去重效率

### 3.3 Connector 插件化设计

```python
class SourceConnector:
    source_type: str
    def fetch(self, config) -> list[RawItem]: ...
    def normalize(self, raw_item: RawItem) -> NormalizedItem: ...
```

**评估**：
- 接口简单清晰，后续扩展成本低
- 符合开闭原则，新增信息源不修改核心代码

## 四、核心功能可行性

### 4.1 数据采集（P0）

**抓取内容**：
1. 用户关注仓库基础信息
2. 用户关注仓库最新 1 条 Release
3. 关键词搜索 Top 5 新仓库

**可行性**：✅ 高
- 都是 GitHub REST API 的标准端点
- 返回结构化 JSON，无需解析 HTML
- 数据量小，处理简单

### 4.2 去重与候选池

**去重策略**：
- URL 去重：`source_type + url`
- 内容去重：`sha256(title + content[:1000])`
- 三层架构：Raw Data → Candidate Pool → Confirmed Knowledge

**可行性**：✅ 高
- 去重逻辑简单，计算开销小
- SQLite 支持唯一索引和 UPSERT
- 候选状态管理（pending/ignored/confirmed）逻辑清晰

### 4.3 个性化排序

**打分公式**：
```
final_score = 兴趣匹配分 * 0.40 + 项目相关分 * 0.30 + 新鲜度分 * 0.15 + GitHub 热度分 * 0.15
```

**可行性**：✅ 高（MVP 阶段）
- 基于显式关键词 + 权重的匹配，逻辑简单可解释
- 无需 embedding 或复杂 ML 模型
- 分数计算规则透明，用户可理解和调试

**后续优化空间**：
- 可引入 LLM 辅助判断语义相似度
- 可基于用户反馈历史动态调整权重

### 4.4 摘要与推荐理由

**MVP 需求**：
- 对 Top N 内容生成摘要
- 生成推荐理由
- 匹配相关兴趣和项目

**可行性**：⚠️ 中等
- 依赖外部 LLM API（OpenAI、Qwen 等）
- 有 API 调用成本和延迟
- 需要设计合理的 Prompt 模板

**建议**：
- MVP 阶段摘要生成可异步执行，失败不影响主流程
- 提供本地模型选项（如 Ollama）降低成本
- Prompt 设计要考虑成本和效果平衡

### 4.5 每日 Markdown 简报

**输出格式**：`data/digests/YYYY-MM-DD.md`

**可行性**：✅ 高
- Markdown 格式简单，无需渲染引擎
- 结构清晰，便于阅读和版本控制
- 可直接在 Git 中追踪历史

- repo 模式实现

- 打分算法实现

## 六、关键风险与控制

### 风险 1：GitHub API 限制

**描述**：API 额度不足、频率限制、服务不可用

**控制措施**：
- 使用 GITHUB_TOKEN 提升额度
- 实现 ETag 条件请求减少无效调用
- 指数退避重试机制
- 全局请求限速
- 单点失败不影响全局任务

### 风险 2：LLM 摘要质量不稳定

**描述**：摘要质量差、API 延迟高、成本高

**控制措施**：
- 摘要失败不阻断主流程
- 支持多种 LLM 提供商（降级方案）
- 提供本地模型选项
- 设计清晰的 Prompt 模板
- 允许用户手动编辑摘要

### 风险 3：信息过载/噪声

**描述**：抓取内容过多、低价值内容污染知识库

**控制措施**：
- 严格的 P0 抓取范围（只抓核心信息）
- 候选池机制（用户确认后入库）
- 每日只展示 Top 5-10
- 支持忽略和反馈

### 风险 4：后续扩展困难

**描述**：架构耦合、难以支持新信息源

**控制措施**：
- Connector 插件化设计
- 统一数据模型 NormalizedItem
- Pipeline 分层解耦
- GitHub 逻辑不污染核心业务

## 八、与现有计划的差异建议

### 8.1 建议补充的点

1. **错误监控和告警**
   - 每日任务失败时需要通知（邮件/日志/Webhook）
   - 建议增加简单的告警机制

2. **数据备份策略**
   - SQLite 单文件容易备份
   - 建议定期备份到 Git 或其他存储

3. **配置热更新**
   - 修改配置文件后无需重启服务
   - 或提供 API 动态修改配置

4. **任务执行日志**
   - 每次抓取任务的详细日志
   - 便于排查问题和审计

5. **测试策略**
   - 单元测试覆盖核心逻辑
   - 集成测试覆盖 GitHub API 调用
   - Mock 测试减少对外部依赖

### 8.2 建议简化的点

1. **API 端点**
   - MVP 阶段 API 可以更少，先聚焦核心功能
   - 部分管理功能可以通过直接修改配置文件实现

2. **数据库表**
   - `request_cache` 表可以延迟到需要时再实现
   - 先用内存字典或文件缓存 ETag

3. **用户反馈**
   - MVP 可以先只做收藏/忽略
   - "不感兴趣"和"稍后阅读"可以后续再增加

## 九、总体评估结论

### 9.1 可行性结论

**综合评估：✅ 高度可行**

理由：
1. **技术成熟**：所有技术选型都是成熟方案，社区支持好
2. **范围可控**：MVP 范围定义清晰，不做过度设计
3. **风险可控**：核心风险都有明确的缓解措施
4. **扩展性好**：架构预留了充分的扩展空间
5. **资源可控**：开发和运行资源需求低

### 9.2 成功关键因素

1. **严格执行 MVP 范围**：不在第一阶段引入知识图谱、向量数据库等复杂功能
2. **LLM 摘要质量**：这是用户感知价值的核心，需要精心打磨 Prompt
3. **稳定性**：定时任务和 API 调用的稳定性直接影响用户体验
4. **可解释性**：推荐理由必须清晰，让用户理解为什么推荐

### 9.3 推荐下一步行动

1. **立即启动**：项目具备立即开发的条件
2. **先搭骨架**：按 Phase 1 先搭建项目结构和配置系统
3. **迭代开发**：每完成一个 Phase 就进行测试和验证
4. **持续反馈**：尽早让真实用户使用，收集反馈调整方向
